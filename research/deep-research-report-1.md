# กลยุทธ์ครอบคลุมข้อมูลสังเคราะห์สำหรับระบบตรวจสอบภาพแบบกำหนดแน่นอน

## ความใช้ได้ของการทดสอบแบบสังเคราะห์ล้วน

**แหล่งหลักและข้อค้นพบแบบหนึ่งบรรทัด** งาน Donut/SynthDoG แสดงชัดว่าข้อมูลสังเคราะห์ช่วยได้มาก แต่ “พอจริงหรือไม่” ขึ้นกับว่า downstream task ใกล้กับการอ่านข้อความและโครงเอกสารเพียงใด; ในการวิเคราะห์ของผู้เขียนเอง ข้อมูลสังเคราะห์เพียงอย่างเดียว “พอ” สำหรับ document IE บางงาน แต่ DocVQA ยังต้องอาศัยภาพจริงเพราะ distribution ใกล้เคียงกันมีผลชัดเจน. งาน DePlot ชี้ว่าการ pretrain/finetune บน chart-to-table แบบสังเคราะห์ช่วยมากกับ benchmark ที่อยู่ในโดเมนเดียวกัน แต่เมื่อออกนอกโดเมน ผู้เขียนต้องทำ OOD test เองเพราะ “มีทรัพยากรสาธารณะที่มีทั้งกราฟและตารางคู่น้อยมาก”; บนตัวอย่าง web charts จาก TaTa เพียง 10 ภาพ DePlot ได้ 78% RMS-F1 และยังพังกับข้อความข้างเคียงและ label/value ที่เชื่อมด้วยลูกศร. งาน GenPlot ให้ข้อสรุปที่ถ่ายโอนได้สูงที่สุดข้อหนึ่ง: ถ้าจะใช้ synthetic เพื่อแทนโลกจริง ความ “หลากหลายเชิงเรนเดอร์และเชิงสไตล์” สำคัญกว่าการทำให้ใกล้โดเมนใดตัวเดียว. งาน SBSFigures ใช้ stage-by-stage synthesis เพื่อเพิ่ม “topic diversity” และ “appearance diversity” ของ figures และรายงานว่า pretraining มีประโยชน์แม้เมื่อมีข้อมูลจริงจำกัด. ในฝั่งเอกสาร DocCreator กับ SynthDoG ต่างตอกย้ำแนวคิดเดียวกันคือ synthetic ใช้ได้เมื่อเก็บ grammar ของ layout, font, text source, appearance และ degradation ได้พอครอบคลุม ไม่ใช่แค่สุ่ม noise ทับภาพเดิม. citeturn9view0turn7view1turn7view2turn8view5turn35view0turn39search0

สำหรับข้ออ้างที่คุณอยากใช้ว่า “ครอบคลุม house styles ของ renderer ได้ ก็ใกล้เคียงกับครอบคลุม distribution ของภาพ tool-generated จริง” วรรณกรรม **สนับสนุนได้บางส่วน แต่ยังไม่ถึงขั้นพิสูจน์**. หลักฐานที่ค่อนข้างแข็งคือ synthetic ช่วยได้มากเมื่อ latent structure ของงานสอดคล้องกับตัวสร้าง เช่น chart-to-table, OCR-free reading, form/layout parsing และเมื่อ generator ครอบคลุม font/layout/style/degradation หลากหลายจริง ๆ; Donut ระบุว่าการ pretrain แบบอ่านข้อความด้วย SynthDoG เพียงพอสำหรับ document IE ในการวิเคราะห์ของผู้เขียน แต่ DocVQA ต้องเห็น real images ด้วย เพราะ distribution ของ IIT-CDIP กับ DocVQA คล้ายกัน. DePlot ก็ย้ำทางอ้อมว่าความสำเร็จบน ChartQA/PlotQA ยังอิงกับความใกล้โดเมนของ style/type ที่อยู่ใน training corpus และเมื่อไปเจอ web charts คนละบ้าน จะเกิด residual gap ชัดเจน. citeturn9view0turn7view1

**วิธีที่ถ่ายโอนได้มากสุดสู่ข้อจำกัดของคุณ** คือเปลี่ยนมุมจาก “synthetic แทนโลกจริงทั้งหมด” เป็น “synthetic ครอบคลุม family ของตัวสร้างจริงที่คุณตั้งใจรองรับ”. ในเชิงปฏิบัติ แปลว่าให้โมเดล generation space ของคุณ factor อย่างน้อยเป็น: renderer family, chart/diagram grammar, typographic system, color/style policy, numeric/text content policy, export path และ degradation path; จากนั้น validate โดย **renderer-held-out / style-held-out / content-held-out / degradation-held-out** แทนการสุ่มรวมทุกอย่างในถังเดียว. ข้อค้นพบจาก GenPlot และ SBSFigures สนับสนุนว่าความหลากหลายเชิง pipeline และ appearance สำคัญมากกว่าการสุ่มค่าภายในสไตล์เดียวซ้ำ ๆ. citeturn2view5turn39search0

**คำเตือนที่วรรณกรรมไม่รองรับ** คืออย่าเหมารวมว่า synthetic benchmark แบบ DVQA/FigureQA/PlotQA รุ่นแรก ๆ เป็นหลักฐานว่า synthetic-only validity เพียงพอสำหรับโลกจริงเสมอไป; งานที่วัด OOD จริงใน chart/document understanding ยังมีน้อย และแม้ในงานที่ดีที่สุดก็ยังมี gap เหลืออยู่กับ adjacent text, leader lines/arrows, tiny text, camera distortions และ figure types ที่ไม่มี latent textual representation ชัดเจน. DePlot พูดตรง ๆ ว่าแนว plot-to-table ยังไม่เหมาะกับ textbook figures หรือภาพที่สร้างด้วย specialized software ซึ่งไม่มี structured representation ชัด. ถ้าระบบคุณจะครอบคลุม free-body diagrams, mechanical plates, flowcharts, circuits และ figures เชิงวิทยาศาสตร์ ความเสี่ยงจาก “ไม่มี latent table ให้แปลง” สูงกว่ากราฟแท่ง/เส้นทั่วไปมาก. citeturn7view1turn7view2

## การออกแบบ coverage ของพื้นที่สร้างข้อมูล

**แหล่งหลักและข้อค้นพบแบบหนึ่งบรรทัด** แนว NIST combinatorial testing รายงานซ้ำ ๆ ว่าความล้มเหลวของซอฟต์แวร์ส่วนใหญ่เกิดจาก interaction ของพารามิเตอร์จำนวนน้อย; ในคู่มือ Practical Combinatorial Testing ระบุว่า coverage ครบทุก 3-way combinations สามารถตรวจพบได้ราว 90%+ ของบั๊กในหลายกรณี และโดยทั่วไปเลือก t อยู่ในช่วง 2–6 ตามความเข้มงวดของงาน. เอกสาร “Input Space Coverage Matters” ขยายผลว่า fault profile ของหลายระบบเพิ่มเร็วมากจนถึงประมาณ t=4 และในบางโดเมน 4–6 way ก็ครอบคลุมเกือบหมด. ACTS รองรับทั้ง base-choice, 1–6 way, constraints และ mixed-strength ซึ่งเหมาะกับ generation spaces ที่มีแกน discrete จำนวนมากและมี dependency ระหว่างค่าต่าง ๆ. ส่วน Latin hypercube sampling ดีมากสำหรับ continuous nuisance variables เพราะมันบังคับให้แต่ละ marginal ถูกครอบคลุมอย่างมีประสิทธิภาพ แต่ไม่ได้รับประกัน discrete interaction coverage แบบ covering arrays. citeturn13view0turn14view2turn22view0turn13view1turn15search14

เมื่อย้ายจาก software configuration ไปสู่ image pipeline แบบของคุณ หลักฐานตรง ๆ ว่า “failure ของ deterministic visual extractors ก็ส่วนใหญ่เป็น 2–3 way” **ยังบาง**. สิ่งที่วรรณกรรมสนับสนุนแน่คือ interaction rule เป็นจุดเริ่มต้นที่ดีมากสำหรับระบบที่มี input model ชัดเจน และ input parameters ถูก discretize เป็น equivalence classes แล้ว แต่ผมไม่พบงานที่ยืนยันแบบหนักแน่นว่ากลไกนี้ครอบคลุมความล้มเหลวของ chart/diagram verifier ได้เหมือนงานซอฟต์แวร์ทั่วไป. ดังนั้นข้อสรุปที่ซื่อสัตย์คือ: ใช้ 2-way เป็น baseline, 3-way เป็น sweet spot เริ่มต้น, และยกระดับเป็น mixed-strength 4-way เฉพาะกลุ่มปัจจัยที่คุณสงสัยว่ามี coupling สูง เช่น `renderer × font system × negative/decimal formatting × anti-aliasing/export DPI` หรือ `diagram grammar × arrowhead style × OCR locale × JPEG path`. citeturn22view0turn13view1

**วิธีที่ถ่ายโอนได้มากสุดสู่ข้อจำกัดของคุณ** คือใช้ sampling แบบผสมสามชั้น ไม่ใช่เลือกวิธีเดียวทั้งระบบ. ชั้นแรกใช้ **stratified sampling** เพื่อประกันว่าทุก renderer family และทุก visual vertical มีส่วนในชุดทดสอบอย่างชัดเจน. ชั้นที่สองใช้ **t-way covering arrays** บนปัจจัยแบบ category/discrete ที่เป็นตัวสร้างรูป เช่น renderer, chart type, legend mode, tick format, color channel, text locale, export backend, degradation family. ชั้นที่สามใช้ **Latin hypercube** เฉพาะ nuisance variables แบบต่อเนื่อง เช่น blur radius, camera angle, contrast loss, JPEG quality, downscale factor, line thickness jitter, label length percentile. แนวนี้สอดคล้องกับข้อเสนอของ NIST ที่ต้อง discretize input space ก่อน แล้วค่อยใช้ combinatorial coverage บนค่าที่มีความหมายเชิง equivalence; ส่วนตัวแปรต่อเนื่องให้ coverage marginal ที่ดีผ่าน LHS จะคุ้มกว่าฝืนบิ้นเป็นหลาย bin แบบหยาบเกินไป. citeturn22view0turn13view1turn15search14

**คำเตือนที่วรรณกรรมไม่รองรับ** คืออย่าใช้ LHS เป็น sampler หลักให้กับ generation space ที่แกนสำคัญส่วนใหญ่เป็น symbolic/discrete; มันไม่ได้ประกัน pairwise/triple-wise interaction และอาจทำให้คุณ “ดูครอบคลุม” ในเชิงค่าเลข แต่พลาด tuple อันตรายของ format/renderer/style ได้ง่าย. อีกด้านหนึ่ง อย่าเชื่อว่าต้องไปถึง 5–6 way ทั้งโลกตั้งแต่วันแรก; NIST เองวาง 2–6 เป็นช่วงใช้งานและทำ mixed-strength เพื่อคุมต้นทุน ไม่ได้ชวนให้ทำ exhaustive high-strength ทุกปัจจัยพร้อมกัน. citeturn13view1turn14view2

## การวัด coverage ให้มีความหมายต่อการพบข้อผิดพลาด

**แหล่งหลักและข้อค้นพบแบบหนึ่งบรรทัด** NIST ให้คำจำกัดความ combinatorial coverage ที่มีความหมายตรงกับ “สัดส่วนของ input-state combinations ที่ได้ทดสอบจริง” และชี้ว่าเป็นตัวชี้วัด test quality ที่เกี่ยวข้องกับ fault detection โดยตรง มากกว่าดูแค่โค้ดถูกรันถึงหรือไม่. เอกสาร Combinatorial Coverage Difference Measurement เพิ่มมุมเทียบสอง test suites ด้วย coverage difference แทนการดู coverage แบบจุดเดียว. ฝั่ง software-testing ทั่วไป งาน Inozemtseva & Holmes พบว่า code coverage มีเพียง low-to-moderate correlation กับ fault detection เมื่อควบคุมขนาด test suite แล้ว ซึ่งแปลตรงตัวว่า coverage สูงไม่ได้เท่ากับตรวจจับบั๊กเก่งเสมอ. ฝั่ง DNN งาน Yan และงาน replication ภายหลังพบว่าพวก neuron/model coverage ไม่มี monotonic relation กับ model quality และ coverage-guided generation ยังแพ้ gradient-based baselines ในหลายการตั้งค่า จึงเป็นตัวอย่างเชิงลบที่ดีว่า “coverage metric ที่ไม่ผูกกับ failure mechanism” อาจพาเราหลงทางได้. citeturn22view1turn20view0turn20view1turn21view1turn24view0

สำหรับ pipeline แบบ deterministic ของคุณ ผมแนะนำให้รายงาน coverage เป็น **สามชั้น**. ชั้นแรกคือ **declared generation-space coverage**: ครอบคลุมแต่ละ stratum เท่าไร, แต่ละ renderer family มี n เท่าไร, และ interaction coverage ระหว่างแกนสำคัญถึงระดับใด. ชั้นที่สองคือ **behavioral/input-partition adequacy**: boundary bins, equivalence classes, rare-format bins และ ambiguous-layout bins ถูกแตะครบหรือไม่. ชั้นที่สามคือ **implementation-side coverage**: branch/condition coverage ของ extractor, parser และ rule engine ใช้เป็น proxy ว่า logic ไหนยังไม่เคยโดน แต่ไม่ยกมันเป็น metric หลักของความเชื่อมั่น. แนวคิดนี้ตรงกับ NIST ที่เสนอให้ใช้ input model + combinatorial coverage เป็นตัววัดชุดทดสอบ และใช้ structural coverage เพื่อตรวจคุณภาพของ requirements-derived tests ไม่ใช่แทนกัน. citeturn22view0turn22view1

ถ้าต้องการ metric เสริมที่สัมพันธ์กับ fault detection มากกว่า branch coverage ผมจะเลือก **mutation/seeded-fault score ใน rule layer** แทน neuron coverage หรือ path-counting แปลก ๆ. วรรณกรรม mutation testing เตือนเหมือนกันว่าความสัมพันธ์กับ real fault detection ก็ไม่ได้แน่นแบบเวทมนตร์ แต่โดยรวมมันเป็นตัวชี้นำการปรับ test suite ได้ดีกว่าแค่ line/branch coverage และสอดคล้องกับงานจาก Google ที่มอง mutation testing เป็นเกณฑ์เข้มแข็งสำหรับ test efficacy. สำหรับระบบของคุณ mutation ไม่จำเป็นต้องแก้ source code แบบเต็มรูป ก็อาจทำเป็น “rule mutations” เช่น พลิก inequality, ปิด tolerance check, เปลี่ยน tick-merge heuristic, ลบ color-identification fallback แล้ววัดว่าชุดทดสอบจับได้กี่ตัว. citeturn19search5turn19search16

**วิธีที่ถ่ายโอนได้มากสุดสู่ข้อจำกัดของคุณ** คือรายงาน coverage ในรูป “coverage ledger” ต่อ vertical: `(ก) stratum coverage, (ข) t-way interaction coverage, (ค) boundary coverage, (ง) branch/rule-mutation coverage, (จ) discovered-failure coverage`. ถ้ามี regression suite สะสม ให้ใช้ coverage-difference ของ NIST เพื่อบอกว่า release ใหม่เพิ่ม coverage ตรงไหน ไม่ใช่แค่จำนวนเคสเพิ่มกี่รูป. citeturn20view0turn22view1

**คำเตือนที่วรรณกรรมไม่รองรับ** คืออย่าตั้ง KPI หลักเป็น code coverage, neuron coverage หรือ “เปอร์เซ็นต์ของ augmentations ที่รันแล้ว” เพราะทั้งประสบการณ์ software testing และผลเชิงลบใน DNN testing บอกตรงกันว่าตัวเลขเชิงกิจกรรมเหล่านี้แยก “เราโดนอะไร” ออกจาก “เราเจอ failure mechanism ที่สำคัญหรือยัง” ไม่ได้ดีพอ. Coverage ต้องผูกกับ input model และ oracle ที่ตรงกับความเสี่ยงจริง มิฉะนั้นจะได้ความสวยของ dashboard มากกว่าความปลอดภัยของระบบ. citeturn20view1turn21view1turn24view0

## การสร้างเคสค้นหาจุดพังด้วย oracle และ fuzzing

**แหล่งหลักและข้อค้นพบแบบหนึ่งบรรทัด** DeepHunter เสนอกรอบ coverage-guided fuzzing ที่ใช้ metamorphic mutation และ feedback เพื่อคัดเลือก seed ต่อไป ส่วน DeepTest ใช้การเปลี่ยนสภาพแวดล้อมที่สมจริงอย่างฝน หมอก แสง เพื่อตรวจพฤติกรรมผิดพลาดจำนวนมากใน DNN ขับรถอัตโนมัติ. ฝั่ง metamorphic testing survey ของ Segura สรุปว่า MT ใช้ได้ดีโดยเฉพาะเมื่อ oracle เต็มรูปแบบหายากหรือแพง. งาน Feedback-Directed Metamorphic Testing แสดงว่าการเลือก metamorphic relations และ source tests แบบ adaptive จาก feedback ระหว่างรัน ช่วยตรวจเจอบั๊กเท่าเดิมด้วยเคสน้อยลงและเวลาน้อยลง. ฝั่ง search-based testing ของ McMinn อธิบายชัดว่าถ้า objective function นิยามได้ดี metaheuristics อย่าง hill climbing, simulated annealing และ evolutionary search จะหาบริเวณที่น่าสงสัยกว่าสุ่มล้วน โดยเฉพาะใน input spaces ใหญ่และมี boundary ยาก. citeturn25search12turn25search9turn27search5turn29search2turn31view0

ภายใต้ข้อจำกัดของคุณ—มี exact oracle จาก renderer, มี round-trip checks, มี invariances และ ensemble disagreement—คุณอยู่ในสถานะที่ **ดีกว่างาน fuzzing ส่วนใหญ่** เพราะไม่ต้องแก้ปัญหา oracle scarcity. ดังนั้นวิธีที่คุ้มสุดไม่ใช่เอา DeepHunter-style infrastructure ทั้งก้อนมาวาง แต่คือ **boundary-first search + adaptive local exploration**. เริ่มจาก sweep ค่าตามขอบของ partition ทุกตัวก่อน เช่น tick spacing ใกล้ switching threshold, font size ที่ OCR เริ่มสั่น, line thickness ใกล้ non-maximum suppression fail, label length ใกล้ collision, JPEG quality ใกล้ quantization jump จากนั้นค่อยใช้ hill climbing หรือ evolutionary perturbation ใน neighborhood ของเคสที่คะแนนหลักฐานเริ่มบางลง เช่น confidence margin ของ geometric fit ต่ำ, OCR disagreement สูง, re-render residual สูง, หรือ pass/fail อยู่ใกล้ threshold. แนวนี้ตรงกับ search-based testing ที่เน้น objective-driven exploration มากกว่าสุ่มทั้งโลก. citeturn31view0

metamorphic testing ในกรณีของคุณควรใช้เป็น **เครื่องขุด failure surfaces** ไม่ใช่เป็นตัวตัดสินความถูกต้องแทน exact oracle. ความสัมพันธ์อย่าง isotropic rescale, uniform brightness shift, crop-safe border expansion, export/import idempotence, หรือ re-render equivalence มีค่าเพราะช่วยยืนยันว่าเคสใดเป็น fragility ของ extractor มากกว่าจะเป็น content ambiguity. ถ้าจะทำ adaptive sampling จริง งาน FDMT ให้ไอเดียที่ถ่ายโอนได้มากคือเก็บสถิติว่า relation ไหนพาไปเจอ violation บ่อยใน stratum ไหน แล้วให้ scheduler อัปเดต prior ของการยิง relation ชุดต่อไป. citeturn29search2turn27search5

**วิธีที่ถ่ายโอนได้มากสุดสู่ข้อจำกัดของคุณ** คือ loop ห้าขั้น: `seed จาก covering array → boundary sweeps → local search รอบเคสเสี่ยง → delta-debug ลดเคสให้มินิมัล → root-cause clustering เพื่อสร้าง regression strata ใหม่`. Delta debugging มีหลักฐานยาวนานว่าช่วยตัด “ส่วนเกิน” ออกจาก failing inputs จนเหลือ minimal failure-inducing cause ส่วนงาน crash deduplication/triage ยุค fuzzing แสดงว่าการจัดกลุ่มตาม root cause สำคัญมากกว่าการนับจำนวนเคสพังดิบ ๆ. สำหรับคุณ สิ่งที่เก็บกลับเข้าชุดทดสอบไม่ควรเป็นแค่ภาพที่พังรูปเดียว แต่ควรเป็น “stratum ใหม่ที่นิยาม boundary ของบั๊ก” เช่น “Mermaid flowchart + orthogonal connectors + text boxes with arrows crossing labels + JPEG<40”. citeturn32search6turn32search7turn32search13

**คำเตือนที่วรรณกรรมไม่รองรับ** คืออย่าหลงกับ coverage-guided fuzzing แบบที่ optimize code coverage อย่างเดียว; ในระบบที่มี exact oracle ราคาถูกและมี generation parameters ชัดเจน การไล่ edge coverage ของโปรแกรมมักแพ้การค้นหาบน semantic parameters ที่ผูกกับสเปกจริง. อีกข้อคืออย่าเอา failure seeds เข้า regression suite แบบตรง ๆ ทั้งหมด เพราะจะ overfit กับ “บั๊กที่เคยเจอ” มากกว่าความเสี่ยงคลาสเดียวกัน; ต้องทำ minimizing + clustering + abstraction เสมอ. citeturn31view0turn32search6turn32search13

## ความสมจริงของ degradation และการคง ground truth

**แหล่งหลักและข้อค้นพบแบบหนึ่งบรรทัด** DocCreator สร้าง synthetic documents พร้อม ground truth และมี degradation models ที่แตะทั้ง ink, paper shape และ background color โดยตั้งเป้าให้ useful สำหรับ precise performance evaluation โดยตรง. Augraphy ถูกออกแบบมาเพื่อเลียนกระบวนการ office จริง เช่น printing, scanning, faxing, dirty machines, ink degradation และ handwritten markings และผู้เขียนระบุชัดว่านอกจากใช้ augment training แล้ว ยังใช้สร้าง challenging test data เพื่อทดสอบ robustness ได้. ocrodeg ให้ชุด random geometric transforms และ page/document degradations ที่เบาและใช้การได้จริงสำหรับ OCR/document pipelines. ฝั่งข้อมูลจริง SmartDoc-QA และ IUPR/งานสำรวจเอกสารจากกล้องยืนยันว่าภาพ document ที่มาจากมือถือหรือกล้องมักมี blur, perspective distortion, non-uniform illumination, shading, curl, border noise และ background interaction ไม่ใช่แค่ JPEG กับ gaussian blur นิดหน่อย. สำหรับ screen-photo เส้นทาง “ถ่ายหน้าจอ” moiré เป็น artifact จริงและมีงานเฉพาะทางศึกษาการเกิดและกำจัดมัน เพราะเป็นปัญหาคนละชั้นกับ blur ปกติ. citeturn35view0turn35view1turn35view3turn34search15turn37view1turn34search9

ดังนั้น degradation harness ที่ถ่ายโอนได้มากสุดสำหรับระบบของคุณควรถูกแบ่งเป็น **สี่สายส่งภาพ** ไม่ใช่ถัง noise เดียว. สายแรกคือ **digital export/render path**: anti-aliasing, alpha compositing, subpixel text rendering, downscale, JPEG/WebP. สายที่สองคือ **office reproduction path**: print-scan, fax/photocopy, dirty toner, paper texture, bleed-through, low ink ซึ่ง Augraphy และ DocCreator รองรับแนวคิดนี้ตรง ๆ. สายที่สามคือ **camera capture path**: perspective/homography, curl/warp, blur, exposure, shading, glare, background leakage, crop and border noise ซึ่ง SmartDoc-QA และ IUPR บอกว่าเป็นปัญหาหลักของ camera-captured document จริง. สายที่สี่คือ **screen-photo and chat path**: resize/recompress pipeline, moiré, temporal screenshot chains และ messenger-style recompression. หลักฐานเรื่อง moiré มีอยู่ แต่หลักฐานสาธารณะเชิงละเอียดสำหรับ compression signature ของแต่ละแอปแชตยังค่อนข้างบางกว่าฝั่ง print/scan และ camera documents. citeturn35view1turn35view0turn34search15turn37view1turn34search9

เรื่อง ground truth geometry เป็นจุดที่งาน document degradation ให้บทเรียนสำคัญมาก: degradation บางประเภทไม่แตะ semantic geometry เลย เช่น JPEG, additive noise, blur, low contrast; แต่บางประเภท **เปลี่ยน geometry ของความจริง** โดยตรง เช่น rotation, anisotropic scaling, perspective transforms, page curl และ local warps. ocrodeg เองโชว์ random geometric transformations และ anisotropy แบบ explicit ส่วนงาน warping generation ระบุชัดว่าพารามิเตอร์เรขาคณิตอย่าง camera angle, position, distance และ surface curvature สามารถเป็น ground truth ของภาพ warped ได้. สำหรับระบบคุณ แปลว่าควรเก็บ transformation stack เป็นวัตถุ first-class แล้ว propagate primitive-level ground truth ด้วย matrix/warp field เดียวกับที่ใช้เรนเดอร์ภาพ ไม่ใช่ generate ภาพเสร็จค่อยคาดเดา geometry ใหม่จากพิกเซล. citeturn35view3turn38view0

**วิธีที่ถ่ายโอนได้มากสุดสู่ข้อจำกัดของคุณ** คือสร้าง degradation API ที่รับทั้ง `scene graph/spec` และ `vector primitives` แล้วส่งออก `(ภาพ degraded, transformed ground truth, lineage metadata)` ทุกครั้ง. ถ้า transform เป็น affine/homography ให้แปลง coordinates ตรง ๆ; ถ้าเป็น non-linear warp ให้เก็บ dense displacement field แล้วใช้มัน warp จุด/เส้น/bbox/anchor text ทั้งหมด. วิธีนี้สอดคล้องกับตรรกะของงาน warping generation และรักษาข้อกำหนด “evidence-backed correctness” ของคุณได้ดีที่สุด. citeturn38view0turn35view0

**คำเตือนที่วรรณกรรมไม่รองรับ** คืออย่าถือว่า blur/downscale/JPEG track เล็ก ๆ แทนโลกจริงแล้ว โดยเฉพาะสำหรับ diagrams ที่มีเส้นบาง, text เล็ก, arrowheads และ leader lines; dataset และ survey ของ camera-based document analysis ยืนยันว่าปัญหาจริงมักมาพร้อม geometric + photometric distortions พร้อมกัน. อีกจุดที่ยังไม่เห็นหลักฐานสาธารณะแรงพอคือการจำลอง “compression signature เฉพาะแอป” ระดับ WhatsApp/Messenger/LINE แบบเชื่อถือได้; ถ้าจะทำ ให้ label ว่าเป็น **engineering proxy** ไม่ใช่ validated simulator. citeturn34search15turn37view1turn34search8

## การใช้ LLM สร้างเนื้อหาเชิงโครงสร้างก่อนเรนเดอร์แบบกำหนดได้

**แหล่งหลักและข้อค้นพบแบบหนึ่งบรรทัด** SBSFigures เสนอ pipeline แบบ stage-by-stage ที่ให้ LLM สร้าง topic/content ก่อน แล้วจึงเรนเดอร์เป็น chart figures พร้อม annotation แบบไม่ต้องติดป้ายด้วยมือ และเน้นชัดเรื่อง topic diversity กับ appearance diversity. Synthesize Step-by-Step ใช้ template data สอน LLM data generator ให้แตกคำถาม reasoning เป็น sub-questions พร้อมให้ Python คำนวณคำตอบ และรายงานว่า accuracy บน ChartQA human-set ขยับจาก 38% เป็น 54% เมื่อใช้ข้อมูลที่ LLM ขยายให้. ChartLlama ใช้ multi-step pipeline ที่แยกการสร้าง tabular data, การสร้าง charts และการสร้าง instruction data ออกจากกัน เพื่อคุม diversity, quality และจำนวน chart/task types ได้ดีกว่า generator รุ่นก่อน. งาน NAACL 2025 เรื่อง auto-feedback สำหรับ chart generation ก็ย้ำว่าปัญหาสำคัญของงานนี้คือข้อมูลชนิด `instruction, data, code` แพงและหายาก จึงต้องอาศัย automated synthesis pipeline. citeturn39search0turn39search5turn39search18turn39search4

สำหรับข้อจำกัดของคุณ วิธีใช้ LLM ที่เข้ากันที่สุดคือให้มันอยู่ที่ **semantic content layer เท่านั้น**: สร้าง data tables, label text, unit choices, locales, titles, plausible domain topics, circuit labels, flowchart node text หรือ mechanical-part callouts แล้วทุกอย่างหลังจากนั้นต้องผ่าน schema validation และ deterministic renderer. งาน SBSFigures กับ ChartLlama สนับสนุนชัดว่าการแยก content generation ออกจาก rendering step ช่วยให้ควบคุมคุณภาพและ diversity ได้ ส่วน Synthesize Step-by-Step แสดงว่าถ้าให้ external tools อย่าง Python เป็นตัวปิดวง reasoning จะลด hallucination ของ content generator ได้ดีกว่าโยนให้ LLM ตอบรวดเดียว. citeturn39search0turn39search5turn39search18

**วิธีที่ถ่ายโอนได้มากสุดสู่ข้อจำกัดของคุณ** คือสร้าง “content realism contract” แบบมี field บังคับ ไม่ใช่ prompt ยาว ๆ แล้วภาวนา. อย่างน้อยควรบังคับ: distribution class ของตัวเลข, สัดส่วนเลขกลม/ไม่กลม, sign patterns, decimals/significant figures, locale ของ thousand/decimal separator, label length percentiles, collision stress level, ontology ของหน่วยและสัญลักษณ์, frequency ของ near-ties/near-overlaps และ frequency ของ special tokens เช่น minus sign, scientific notation, percent, SI prefix. ให้ LLM สร้าง candidate content แต่ต้องมี deterministic validators เช็ก schema, unit consistency, lexical diversity, banned triviality และ collision likelihood ก่อนเรนเดอร์. แนวคิด “เครื่องมือภายนอกปิดวง” จาก Synthesize Step-by-Step ควรถูกนำมาใช้เต็ม ๆ กับ numeric realism ของคุณ. citeturn39search5

**คำเตือนที่วรรณกรรมไม่รองรับ** คือหลักฐานเชิงตรงที่บอกว่า “label length”, “locale number formats”, “non-round distributions” หรือ “topology realism” ตัวไหนเพิ่ม downstream difficulty เท่าไร **ยังไม่หนาพอ** ใน chart literature ปัจจุบัน. งานเด่นส่วนมากพิสูจน์ว่า multi-stage synthesis และ LLM-assisted generation ทำให้ dataset หลากหลายขึ้นและช่วยโมเดลได้ แต่ไม่ได้ทำ ablation แยกละเอียดว่าความยากเพิ่มเพราะ factor ใด factor หนึ่งโดยเฉพาะ. เพราะฉะนั้น ถ้าคุณจะนิยาม taxonomy ของ chart content realism สำหรับ test-suite design นั่นจะเป็น contribution เชิงระบบของคุณเองมากกว่าจะเป็นสิ่งที่ literature ปิดประเด็นไว้แล้ว. citeturn39search0turn39search18turn39search4

## การประเมิน selective prediction และการรายงานผลอย่างซื่อสัตย์

**แหล่งหลักและข้อค้นพบแบบหนึ่งบรรทัด** Geifman & El-Yaniv วางนิยามมาตรฐานของ selective risk และ coverage พร้อม risk-coverage curve เป็น performance profile หลักของระบบที่มี reject option. งานของ Ding et al. ชี้ว่า AURC เป็น metric ที่น่าเชื่อถือกว่าพวก AUROC/AUPR สำหรับ selective prediction เพราะมันสอดคล้องกับ tradeoff ของ “ความเสี่ยงบนส่วนที่ตัดสิน” กับ “coverage” โดยตรง. งานใหม่กว่าของ Traub et al. เตือนว่าการประเมิน selective systems ยังมีข้อผิดพลาดเชิง metric ได้ และเสนอ AUGRC เป็นตัวเลือกเมื่ออยากได้การตีความแบบ average risk of undetected failures ที่ยืดหยุ่นขึ้น. ฝั่งสถิติ interval estimation สำหรับ binomial สรุปค่อนข้างตรงกันว่า Wald/normal interval ไม่น่าใช้กับ sample เล็กหรือกรณีสัดส่วนสุดขั้ว; Wilson และ exact Clopper–Pearson ปลอดภัยกว่า. เมื่อไม่พบความล้มเหลวเลย rule-of-three ให้ upper 95% bound แบบคร่าว ๆ ที่ `3/n`, ซึ่งเป็นวิธีสื่อสารที่แพทย์และงาน safety ใช้กันแพร่หลาย. citeturn41search4turn40search8turn40search12turn41search5turn41search0turn41search9

สำหรับ validation report ของคุณ ผมแนะนำให้แยก metric เป็นสามบรรทัดเสมอ: **coverage** = สัดส่วนเคสที่ระบบตัดสิน pass/fail; **selective risk** = error rate บนเคสที่ตัดสินแล้วเท่านั้น; และ **abstention profile** = สัดส่วน needs_review พร้อมเหตุผล rule-level ที่เด่นที่สุด. ในระดับภาพรวมให้วาด risk-coverage curve และรายงาน AURC; ในระดับ business claim ให้ report เป็น “accuracy-on-decided at X% coverage per stratum” พร้อมช่วงความเชื่อมั่น. ถ้าคุณต้องการความเข้มงวดสูง แบบ “เกือบ 100% correctness on decided cases” ควรรายงาน failure rate มากกว่า accuracy เพราะมันตรงกับภาษาความเสี่ยงกว่า เช่น “0/87 failures observed on decided bar-chart cases; exact 95% upper bound on failure rate = …”. หลักคิดนี้สอดคล้องกับทั้ง selective-classification literature และสถิติ binomial แบบอนุรักษ์นิยม. citeturn41search4turn40search8turn41search5turn41search0

**วิธีที่ถ่ายโอนได้มากสุดสู่ข้อจำกัดของคุณ** คือรายงานผลแบบต่อ stratum และต่อ decision type ดังนี้: จำนวนเคสทั้งหมด, จำนวน decided, coverage พร้อม Wilson interval, จำนวน failures ใน decided, selective risk พร้อม exact binomial หรือ Wilson interval, และถ้า observed failures = 0 ให้แนบ rule-of-three หรือ exact upper bound ทันที. สำหรับ strata เล็กมาก อย่า aggregate จนดูสวยแต่หลอก; ให้ยอมรับว่าช่วงความเชื่อมั่นกว้างและใช้มันตัดสินว่าจะลงทุนเพิ่มใน strata ไหน. ในเชิงปฏิบัติ Wilson interval เหมาะกับ coverage/accuracy ทั่วไป ส่วน Clopper–Pearson หรืออย่างน้อย rule-of-three เหมาะมากกับกรณี zero failures ที่คุณน่าจะเจอบ่อยเมื่อตั้ง threshold ให้อนุรักษ์นิยม. citeturn41search5turn41search0turn41search9

**คำเตือนที่วรรณกรรมไม่รองรับ** คือห้ามเขียนว่า “100% accuracy” เฉย ๆ เมื่อ n เล็กและมี abstention เยอะ เพราะมันปะปนสองคำถามคนละข้อ: “ตัดสินได้กี่เคส” กับ “ตัดสินแล้วพลาดกี่เคส”. อีกอย่างหนึ่ง AURC ดีมากสำหรับ summary แต่ไม่ควรใช้แทนตารางต่อ stratum; metric เดียวซ่อน failure pockets ได้ง่าย โดยเฉพาะในระบบกันพลาดที่ยอม abstain หนัก ๆ. citeturn41search4turn40search8turn40search12

## การสร้างแผนภาพเชิงโครงสร้างและลำดับการลงทุนที่ควรทำ

**แหล่งหลักและข้อค้นพบแบบหนึ่งบรรทัด** งาน graph grammars ของ Merrell แสดงว่ากราฟไวยากรณ์สามารถใช้ procedural generation สร้างรูปร่างใหม่จากตัวอย่างได้ โดยรักษาความคล้ายเชิงโครงสร้างในระดับ local rules. งานด้าน process-model generation ใช้ graph grammars เพื่อสร้าง process models และ event logs อย่างเป็นระบบ ซึ่งถ่ายโอนได้กับ flowcharts/workflows มากกว่าที่เห็นในชื่อกระดาษ. งานใหม่ใน procedural content generation ก็ยังยืนยันว่ากราฟไวยากรณ์เหมาะกับการสร้างโครงสร้างที่ต้องคุม constraint ไม่ให้ผิดกติกา. ฝั่ง evaluation ของ graph generation งาน ICLR 2022 ชี้ว่า metric สำหรับกราฟควรดูเป็นครอบครัว เช่น graph statistics, graph kernels และ metric เชิง distribution ไม่ใช่ตัวเดียว. งาน motif-paths สนับสนุนมุมมองที่ว่า motif instances เป็น higher-order units ที่ใช้วัดโครงสร้างได้มีความหมายกว่าดู edge count อย่างเดียว. citeturn40search2turn40search21turn40search6turn40search22turn40search15

สำหรับ use case ของคุณ วิธีที่ถ่ายโอนได้มากสุดคือ **สร้าง diagram จาก underlying typed graph/constraint system ก่อน แล้วค่อย deterministic layout/render**. Flowcharts ให้ใช้ attributed graph grammar ที่มีกฎสำหรับ start/end, process, decision, fork/join, loop-back และ cross-link พร้อม constraints เรื่อง reachability และ acyclicity เฉพาะบางโหมด. Circuit schematics ให้ใช้ netlist grammar + motif library เช่น series/parallel, bridge, ladder, feedback, Wheatstone-like structures, op-amp macro blocks แล้วให้ layout engine สร้าง symbol placement ภายหลัง. CAD-like mechanical plates ให้ใช้ parametric sketch grammar มากกว่า image augmentation: hole arrays, slots, chamfers, dimension chains, GD&T-like callouts, symmetric/antisymmetric plates, tolerance annotations. สิ่งสำคัญคือ production history ของ grammar จะกลายเป็น **exact structural ground truth** ของคุณโดยธรรมชาติ. citeturn40search2turn40search21turn40search6

topological coverage สำหรับ structured diagrams ยังไม่มีมาตรฐานเดียวที่ literatures ปิดบัญชีไว้แล้ว ดังนั้นคุณควรกำหนดเองอย่างเปิดเผย. สิ่งที่คุ้มจริงคือรายงานอย่างน้อย: **grammar-rule coverage**, **motif coverage**, **degree-pattern coverage**, **path-length bins**, **symmetry bins**, **crossing/planarity bins**, และ **constraint-boundary coverage**. ฝั่ง evaluation สำหรับ graph generative models สนับสนุนการใช้ graph statistics + kernels เป็น family ของตัววัด ไม่ใช่ scalar เดียว; ฝั่ง motif-path work ช่วยให้ justification ว่าการนับ higher-order motifs เป็นหน่วย coverage มีเหตุผลเชิงโครงสร้าง. citeturn40search22turn40search15

**คำเตือนที่วรรณกรรมไม่รองรับ** คือผมไม่พบ framework มาตรฐานที่ “พร้อมใช้” สำหรับ generating educational circuits/flowcharts/mechanical plates พร้อม topological coverage metric ครบชุดในแบบที่คุณต้องการ; หลักฐานแข็งแรงกว่ามาในระดับ general graph grammars, process-model generation และ procedural modeling. ดังนั้นส่วนนี้คุณน่าจะต้องประกอบเองจาก graph grammar + deterministic layout + coverage taxonomy เฉพาะโดเมน และนี่ไม่ใช่สัญญาณแย่—มันแปลว่า design space ยังเปิดให้ทำของที่ดีกว่าของเดิมได้. citeturn40search2turn40search21turn40search22

**ลำดับการลงทุนที่แนะนำสำหรับนักพัฒนาคนเดียว**
